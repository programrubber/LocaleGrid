LocaleGrid 실제 구현 계획

PLAN.md의 MVP 요구사항을 실제 PyCharm 플러그인 구현 단계로 정리한 문서

목차 목표 현재 상태 결정 필요 사항 다국어 표시 대안 구현 단계 우선순위 완료 기준

목표

PLAN.md의 MVP를 실제 동작 가능한 PyCharm 플러그인으로 완성한다.

핵심 목표는 locales/{locale}/{category}.json 구조의 JSON 파일을 열었을 때, JSON 원본 탭 옆 다국어 에디터 탭에서 다국어 값을 비교, 수정, 검증, 저장할 수 있게 만드는 것이다.

현재 상태

이미 구성된 기반

  • 기존 secui-version-checker 코드는 제거됨
  • 플러그인 ID와 이름은 LocaleGrid로 변경됨
  • FileEditorProvider 기반 커스텀 에디터 구조 추가됨
  • locales/{locale}/{category}.json 경로 인식 로직 추가됨
  • dot path flatten/unflatten 기본 로직 추가됨
  • locale별 key union 병합 로직 추가됨
  • Swing 기반 그리드 UI 초안 추가됨
  • 프로젝트 설정 UI 초안 추가됨

보완 필요

  • PLAN의 확정 사항은 Kotlin이지만 현재 구현은 Java임
  • 저장 전 변경 요약이 아직 line diff 수준이 아님
  • rename 추적이 명확하지 않음
  • readonly/unsupported value 표시가 단순함
  • 구역 표시용 key UI가 아직 충분히 구분되지 않음
  • 단위 테스트가 없음
  • 샌드박스 프로젝트 검증 데이터가 없음

결정 필요 사항

1. 구현 언어

권장 결정: MVP 안정화까지는 현재 Java 구조를 유지하고, 기능이 고정된 뒤 필요하면 Kotlin으로 점진 이전한다.

이미 에디터, 모델, 로드/저장 구조가 Java로 만들어져 있어 지금 Kotlin 전환을 먼저 하면 기능 검증보다 구조 변경 비용이 커진다. JetBrains Platform API는 Java/Kotlin 모두 사용 가능하다.

2. 다국어 표시 방식

한국어, 영어, 일본어, 베트남어를 모두 한 줄에 컬럼으로 표시하면 한 행이 과도하게 길어지고 가독성이 떨어진다.

권장 결정: 기본 화면은 상태 + key + 언어별 컬럼 중심으로 표시한다. 번역 묶음은 마지막 컬럼으로 두되 기본 숨김 처리하고, 테이블 위 체크박스로 번역 묶음과 각 언어 컬럼을 show/hide 할 수 있게 한다. 상태는 key 왼쪽의 별도 고정폭 컬럼에 배지로 표시하고, 정상 행은 배지를 표시하지 않는다.

다국어 표시 대안

PLAN의 key union 그리드 구조는 유지하되, 장문 다국어 값을 가로 컬럼에 모두 넣지 않는다. 사용자는 필요한 언어별 컬럼을 켜서 비교하고, 필요할 때만 마지막 번역 묶음 컬럼을 켤 수 있다. 선택 행 상세 패널에서는 locale별 값을 바로 수정한다.

구현 단계

  1. 모델 정리

    TranslationTable, LocaleGridRow, LocaleValue에 기준 파일, locale 순서, row 상태, rename 추적, 파일 존재 여부를 명확히 저장한다.

  2. JSON 처리 로직 고도화

    flatten 순서 보존, unflatten 충돌 검증, comment key 보존, unsupported value metadata, 저장 전 메모리 직렬화 검증을 강화한다.

  3. 검증 체계 정리

    error와 warning을 분리하고, Diagnostic에 locale, file, rowKey, column 정보를 추가한다.

  4. 그리드 UI 개선

    기본 그리드는 상태, key, 언어별 컬럼으로 구성한다. 자동 locale 순서는 ko, en, jp, vi를 우선한다. 번역 묶음은 마지막 컬럼이지만 기본 hide다. 상태는 key 왼쪽의 고정폭 컬럼에 표시하고, 번역 묶음과 언어별 컬럼은 체크박스로 show/hide 한다. 모든 locale 값은 상세 패널에서 세로로 표시하고 바로 수정한다.

  5. key 편집 기능 완성

    Add Row, Rename Key, Delete Row를 PLAN 흐름대로 구현하고 저장 전 요약에서 추가/수정/rename/삭제 수를 구분한다.

  6. 저장 흐름 완성

    Validate, warning 확인, 변경 요약, 누락 locale 파일 생성 확인, 파일 쓰기, VFS refresh, reload 순서로 안정화한다.

  7. 프로젝트 설정 완성

    locales root, manual locales, comment key 목록, JSON indent 설정을 그리드와 상세 패널에 반영한다.

  8. 테스트 추가

    JSON flatten/unflatten, dot path 검증, key union, locale 탐지, add/rename/delete 반영, 저장용 JSON 생성을 단위 테스트로 검증한다.

권장 구현 순서

  1. Java 유지 결정 후 현재 코드 안정화
  2. TranslationTable, LocaleGridRow, LocaleValue 모델 보강
  3. Diagnostic 구조 확장
  4. 저장 전 메모리 검증 강화
  5. 그리드 UI를 상태 배지 컬럼 + key + 언어별 컬럼 + 번역 묶음 마지막 컬럼 기본 hide + 상세 패널 구조로 개선
  6. Add/Rename/Delete 저장 반영 완성
  7. 단위 테스트 추가
  8. 샌드박스 프로젝트 수동 검증

최종 MVP 완료 기준

  • JSON 파일을 열면 JSON / 다국어 에디터 탭이 표시된다.
  • 같은 category의 locale JSON 파일들이 하나의 테이블로 병합된다.
  • 다국어 value를 한 화면에서 비교하고 수정할 수 있다.
  • 긴 다국어 문장이 한 줄에 과밀하게 표시되지 않는다.
  • Add, Rename, Delete, Save, Refresh, Validate가 동작한다.
  • 빈 value는 warning으로 표시되고 저장은 가능하다.
  • 중복 key와 dot path 충돌은 error로 표시되고 저장이 차단된다.
  • 누락 locale 파일은 저장 시 사용자 확인 후 생성된다.
  • 저장 전 변경 요약이 표시된다.
  • 핵심 JSON 처리 로직 단위 테스트가 존재한다.
